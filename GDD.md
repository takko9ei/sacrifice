> 引擎：Godot 4.2+（GDScript）
> 定位：Game Jam 可交付 Demo，架构为长期扩展预留接口

## 0. 文档说明

本文件是《牺牲》正式开发的**唯一权威依据**。所有关卡、美术、音频、程序工作以本文为准；与本文冲突的口头约定一律以本文为准，如需变更请先修订本文档再实现。

阅读顺序建议：程序先读 §2、§3、§8；关卡设计读 §2、§5、§9.1；美术读 §4、§6、§9.2；手感负责人读 §3、§9.3。

**术语速查**：
- **概念（Concept）**：一个可被牺牲的抽象对象，用字符串 id 标识（如 `gravity`、`blue`、`hud`）。
- **牺牲（Sacrifice）**：使某个概念的效果生效。分"可逆"与"永久"两种。
- **槽位（Slot）**：同一时刻能同时激活的概念数量上限。初始为 1，"双槽"升级后为 2。
- **祭坛（Altar）**：改变牺牲状态的触发器。
- **信号总线**：`Sacrifice` 单例发出的信号集合，是所有系统间通信的唯一渠道。

## 1. 游戏概述

### 1.1 一句话定位
一个"减法"银河城：玩家不靠获得能力，而靠**牺牲掉某个抽象概念**来通过原本过不去的地方。

### 1.2 核心概念
常规银河城做加法（打败 Boss→获得二段跳→新区域解锁）。本作做减法：玩家不断从世界里删除概念。关键在于**每次牺牲都是双刃的**——关掉一个东西的同时打开另一条路。因此它仍然能套进银河城"以前过不去、现在过得去"的基本回路。

### 1.3 核心设计支柱
以下三条支柱在任何取舍中优先级最高，冲突时以靠前者为准：

1. **每次牺牲都必须双刃**：任何"只有代价、没有通路回报"的牺牲都不合格，必须砍掉或重设计。
2. **可读性优先**：玩家任何一秒都能判断"我现在牺牲了什么、还有几个槽、当前哪些东西是实/虚"。
3. **减法带来的失控感是主题**：世界随游戏进程越来越空、越来越怪，通关=把世界拆到只剩出口。

### 1.4 平台与范围
- 平台：PC（键盘）。
- 引擎：Godot 4.2+，GDScript。
- 范围：一个 5 分钟可通关的 Demo，包含一套完整银河城回路。**范围纪律见 §10.1，未列入 MVP 的内容 jam 期间不做。**

### 1.5 调性
祭坛与世界以一种事务性的、不带情绪的方式处理"牺牲"这件事。调性通过**世界内的细节**传达（祭坛的反应、极短的文字、界面的解体），而非通过说明或旁白。实现原则：**少而克制，让玩家自己撞见，不要点破。**

## 2. 核心机制：牺牲系统

### 2.1 总原则
每次牺牲同时产生两个结果：失去某概念的原有功能 + 获得一种新的通过能力。设计任何牺牲时都必须同时明确这两侧。

### 2.2 两层结构
牺牲分为两层，这是让机制既可玩又能在 jam 周期内做完的关键。

**第一层——可逆工具（玩家的移动系统）。** 少数几个概念作为随身工具，可在任意时刻**免费、瞬时**切换（数字键）。因为切换免费，玩家永不软锁，半空也能切回。demo 中为 `gravity` 与 `blue`。

**第二层——永久牺牲（世界的脊柱）。** 少数不可逆的、改变全局或推进主线的牺牲，通过祭坛触发。负责情绪重量与"世界越拆越空"的递进。demo 中为 `jump`（换双槽的代价）、`hud`（界面坍塌，选做）、以及结局的第四面墙。

### 2.3 单槽约束
核心规则：**同一时刻只能激活 `max_slots` 个概念**（初始 1）。切换到新概念时，若已满槽，则自动关闭**最早激活**的那个。

这条规则本身就是免费的谜题发生器：若一段路"既要 A 又要 B"，单槽下就过不去，玩家必须自行意识到"需要同时开两个"——这一刻银河城的"门"就立起来了。

### 2.4 双槽升级
中期通过祭坛获得"双槽"（`max_slots` 1→2），可同时激活两个概念。这是本作的"二段跳"：拿到它后回到先前因单槽不够而过不去的地方，一次性打通，闭环完成。双槽本身用一次永久牺牲换取（见 §2.6 的 `jump`）。

### 2.5 概念库总表

| id | 类型 | 失去 | 换来 | 对应门 | 代价 | 收录状态 |
|---|---|---|---|---|---|---|
| `gravity` | 可逆 | 下落，改为向上"坠" | 走天花板、向上过竖井 | 无踏板的竖井 | 脚下地面失去意义，无法主动向下 | **MVP** |
| `blue` | 可逆 | 蓝色物体的实体性 | 穿过蓝墙/蓝门 | 蓝色屏障 | 蓝平台站不住、蓝开关按不动 | **MVP** |
| `jump` | 永久 | 跳跃能力 | 换取"双槽"升级 | —（作为升级价格） | 失去常规垂直移动手段 | **MVP** |
| `hud` | 永久 | 看到自身牺牲状态的能力 | 牺牲瞬间，HUD 图标坠入关卡、砸成固定的实体平台，可踩以够到原本够不到之处 | 够不到的高台/缺口 | 需靠记忆与画面反馈判断当前状态 | 选做（Post-MVP） |
| `fourthwall` | 永久 | 游戏作为"游戏"的完整性 | "出去"（结局） | 最终出口 | 游戏结束 | **MVP（结局）** |
| `pause` | 永久 | 暂停能力 | 双槽的低成本替代价格 | —（可选升级价格） | 无法暂停 | 选做（备选价格） |
| `friction` | 可逆 | 摩擦力（全世界变冰面） | 冲坡、滑过长断崖、钻矮缝 | 长斜坡/连续窄缝 | 精细平台跳变难，易冲过头 | 扩展储备 |
| `time` | 可逆 | 按时间运作之物停止 | 踩定格子弹/敌人作踏板 | 弹幕房间 | 你需要的移动平台也会停 | 扩展储备 |
| `sound` | 可逆 | 全场静音 | 让靠声音感知的敌人失效 | 声控/声觉敌人区 | 声控机关无法触发 | 扩展储备 |

> "扩展储备"= 机制成立但不在本 Demo 实现，作为后续内容池。"选做"= 若 MVP 提前完成且有余力，按 §10.1 顺序追加。

### 2.6 各概念详细规格

**`gravity`（重力）——可逆，MVP**
- 效果：重力方向翻转，玩家向上加速直至落到天花板；角色精灵上下翻转以示"站在新表面"。
- `up_direction` 随之反向，使 `is_on_floor()`、贴地吸附在天花板上照常工作。
- 双刃：可上天花板、向上过竖井；但脚下地面失效，无法主动向下。
- 空中操控保持不变，随时可切回。

**`blue`（蓝色实体性）——可逆，MVP**
- 效果：所有标记为蓝的碰撞体失去碰撞（可穿过），同时变半透明作为可读性提示。
- 双刃：能穿蓝墙；但蓝平台站不住、蓝开关按不动。恢复后全部复原，完全可逆。
- 可扩展为红、黄等多套彩色门（复用同一模板脚本，仅改 id）。

**`jump`（跳跃）——永久，MVP**
- 触发点：双槽神龛，作为"双槽"升级的价格。
- 效果：永久禁用跳跃输入。触发后不可恢复。
- 设计约束：见 §5.4，触发点之后到通关必须全程不需要跳。

**`hud`（牺牲指示器界面）——永久，选做**
- 效果：牺牲瞬间，屏幕角落的 HUD 图标坠落进世界空间，在预设位置砸成若干一次性的静态实体平台。此后状态 UI 永久消失。
- 双刃换来：这几块"界面残骸"平台让玩家踩着够到原本无法到达的地方（凭空增加几何，与 blue 的"移除几何"正好相反）。
- 代价：永久失去状态显示，此后只能靠记忆判断当前激活了什么。
- 使用范围：作为一处一次性 set-piece，平台位置由关卡设计者预设，不做成可复用系统。

**`fourthwall`（第四面墙／游戏本身）——永久，MVP（结局）**
- 触发点：最终出口的最后一次牺牲。
- 效果：把"这是一个游戏"这件事交出去，换来"出去"。表现为界面逐个解体（HUD 掉落、画面元素消解，最终黑屏）。
- 实现约束：**纯视觉表现**，不真正操作操作系统窗口。做出"界面在解体"的观感即可。

**`pause`（暂停菜单）——永久，选做（备选价格）**
- 作用：作为"双槽"升级的低成本替代价格，面向怕难的玩家路线。代价极低（jam 本不需要暂停）。
- 若采用此价格，则 §5.4 的"通关不需要跳"约束可放宽（因为跳跃被保留）。**注意：`jump` 价格与 `pause` 价格二选一，不要同时对同一升级收两份价。**

## 3. 玩家与操作手感

### 3.1 控制方案
| 动作 | 输入动作名 | 默认键 |
|---|---|---|
| 左右移动 | `move_left` / `move_right` | A/D 或 ←/→ |
| 跳跃 | `jump` | Space |
| 献祭重力 | `sacrifice_gravity` | 1 |
| 献祭蓝 | `sacrifice_blue` | 2 |
| 重开（demo） | `restart` | R |

键位可任意改，不影响手感。新增概念时在 `SacrificeInput.bindings` 加一行并绑键。

### 3.2 移动模型
采用分离重力（split-gravity）模型以获得良好手感：给定跳跃高度 h、上升耗时 t_up、下落耗时 t_down，则上升重力 = 2h/t_up²，下落重力 = 2h/t_down²，起跳速度 = 2h/t_up。t_down 小于 t_up 时下落更利落。"上升/下落"的判定相对于当前重力方向，因此翻转后手感规律不变。

### 3.3 PlayerConfig 参数表（完整）
所有手感数值存于 `PlayerConfig` 资源（.tres），代码零硬编码，运行时改即时生效。

| 参数 | 默认值 | 含义 | 调整方向 |
|---|---|---|---|
| `move_speed` | 180 | 水平最高速（px/s） | 大=更快 |
| `ground_acceleration` | 1800 | 地面加速到最高速 | 大=更跟手 |
| `ground_friction` | 2200 | 地面停下 | 大=更急停 |
| `air_acceleration` | 1200 | 空中转向力 | 大=空中更可控 |
| `air_friction` | 400 | 空中阻力 | 小=更飘 |
| `jump_height` | 64 | 跳跃峰值高度（px） | 大=跳更高 |
| `time_to_peak` | 0.38 | 起跳到最高点耗时（s） | 小=上升更快 |
| `time_to_fall` | 0.30 | 最高点回落耗时（s） | 小=下落更利落 |
| `jump_cut_multiplier` | 0.45 | 松键时保留的上升速度比例 | 小=短按跳更矮 |
| `max_fall_speed` | 600 | 沿重力方向的终端速度 | 大=可下落更快 |
| `coyote_time` | 0.10 | 离地后仍可跳的宽限（s） | 大=更宽容 |
| `jump_buffer_time` | 0.10 | 落地前预输入跳跃窗口（s） | 大=更宽容 |

### 3.4 辅助手感
必须实现：土狼时间（coyote）、跳跃缓冲（buffer）、可变跳跃高度（松键截断）。三者是平台跳"手感好"的底线，不可省略。

### 3.5 重力翻转的行为细节
- 翻转瞬间不改变水平速度，仅改变重力方向与 `up_direction`。
- 翻转时若玩家在空中，继续沿新重力加速；若在地面，则开始向另一面坠落。
- 精灵 `flip_v` 由控制器自动设置（需美术把 `sprite_path` 指向带 `flip_v` 的节点，见 §9.2）。

## 4. UI / UX 设计

### 4.1 必要 UI
本作**唯一"缺了就玩不了"的 UI 是牺牲状态指示器**。其余均为锦上添花，MVP 从简。

### 4.2 牺牲状态指示器（必做）
- 内容：槽位数（N 个格子）、每个已解锁概念一个图标。
- 状态可视：未解锁=灰/不显示；已解锁未激活=常态；激活中=高亮。
- Demo 阶段可用文字版（现有 HUD 已实现 Slots/Active/Unlocked），美术就绪后替换为图标，机制代码不动。

### 4.3 切换反馈（建议做）
切换牺牲的瞬间，除图标变化外，给一个极短的全屏反馈（画面轻微染色 / 一行字弹现），让"世界规则变了"有分量。属 game feel，非信息 UI。

### 4.4 控制提示（建议做，diegetic 优先）
首次进入某祭坛时提示对应操作。优先做成世界内提示（祭坛旁的牌子），而非独立 UI 面板——更省工、更契合调性。

### 4.5 明确不做的 UI
血条/生命值（无战斗伤害）、小地图（demo 仅几屏）、正经暂停菜单（见 §2.6 `pause`，它是牺牲对象而非功能）、主菜单/设置（jam 版直接进游戏，最多一个标题画面）。

### 4.6 UI 作为牺牲对象
纳入三个（`hud`、`fourthwall`、`pause`），规格见 §2.6。核心设计判断：**UI 类牺牲一律走"永久牺牲"层，不做随身可逆工具**——因为 UI 频繁闪灭会令人烦躁，且其不可逆的失去感恰好落在"世界越拆越空"的主线情绪上。实现上它们不是新系统，只是让 HUD 脚本像 `BlueObject` 监听 `blue` 一样去监听 `hud`/`fourthwall` 信号并隐藏/解体对应元素（见 §8.6）。

**未纳入的 UI 牺牲及理由**：生命/失败状态牺牲（本作无 fail state，无处安放）、准星/瞄准 UI 牺牲（无指向性操作）。二者列为储备，若日后加入相应系统再考虑。

## 5. 关卡设计

### 5.1 设计原则
每个新机制按"**教学 → 测试 → 看见门槛 → 拿钥匙 → 打通**"的节奏引入。地图刻意做小，回头路短。任何谜题的解法必须能被玩家从"失去→换来"的双刃关系推理出来。

### 5.2 演示关完整流程（权威关卡目标）
以下 6 房间流程为 Demo 的关卡目标。正式关卡在编辑器里用 TileMap + 放置祭坛/机关场景实现（见 §9.1）；当前代码测试台（`demo_level.gd`）实现的是本流程的压缩线性版，用于即刻验证机制。

```
                 [R6  出口·结局(fourthwall)]
                      │（向上）
                 [R5  蓝色竖井]  ← 核心门槛，需 gravity+blue 同时
                      │
   [R4 双槽神龛(牺牲jump)]──[R3  蓝色区·蓝祭坛]
                      │
                 [R2  重力竖井]
                      │
                 [R1  起点·重力祭坛]
```

- **R1 起点（教 gravity，~45s）**：平地，右侧有跳不上去的高台。走到重力祭坛→学会献祭重力→翻转"坠"到高台（现为天花板侧落脚点）→切回落到出口。只教一件事：献祭重力把"上"变为可达。
- **R2 重力竖井（测试 gravity，~30s）**：纯竖直通道，靠切换重力在地板/天花板间交替爬到顶部出口。无敌人。
- **R3 蓝色区+蓝祭坛（教 blue 并看见门槛，~60s）**：进门一道蓝墙→蓝祭坛→学会献祭蓝、穿墙。关键：本房间内能看见 R5 蓝色竖井入口——那里既要向上（开 gravity）又要穿蓝刺（开 blue），单槽下二选一都过不去。玩家由此意识到需要双槽。R3 同时通往 R4。
- **R4 双槽神龛（拿双槽，~60s）**：平地可达的死路房间（**故意不需要跳**）。神龛用永久牺牲 `jump` 换双槽（叠放两个祭坛：`SET_SLOTS 2` + `PERMANENT_SACRIFICE jump`，一次进入同时触发）。拿到后返回 R3。
- **R5 蓝色竖井（打通核心门槛，~45s）**：同时开 gravity+blue，向上"坠"穿过整条布满蓝刺的竖井到顶。此处即"之前过不去、现在用新能力打通"，闭环完成。
- **R6 出口·结局（~40s）**：封死的出口 + 最终祭坛，索要 `fourthwall` 牺牲，界面解体，Demo 结束。

一周目约 5 分钟（新手 7–8 分钟）。

### 5.3 门与钥匙对应表
| 门 / 障碍 | 所在 | 所需牺牲 | 所需槽位 |
|---|---|---|---|
| 跳不上的高台 | R1 | `gravity` | 1 |
| 竖直通道 | R2 | `gravity`（反复切换） | 1 |
| 蓝墙 | R3 | `blue` | 1 |
| 蓝色竖井（向上+蓝刺） | R5 | `gravity` + `blue` | 2 |
| 最终出口 | R6 | `fourthwall` | — |

### 5.4 关卡硬约束（必须人工验证）
若双槽以永久牺牲 `jump` 换取（默认方案），则 **R4 → R5 → R6 全程不得需要跳**（用重力翻转覆盖所有垂直移动）。R1–R3 用过跳无妨，因玩家不再往回走。**验证方法：牺牲跳跃后逐房间实走一遍确认可通关。** 若改用 `pause` 价格（§2.6），此约束取消。

### 5.5 界面坍塌 set-piece（`hud` 牺牲，选做）
一处专门的落差点：玩家面前有一段够不到的高台/缺口。旁边祭坛 `PERMANENT_SACRIFICE hud`。触发后 HUD 图标坠入世界、在设计好的位置砸成几块实体平台，玩家借此登上高台通过。回报清晰：牺牲前过不去、牺牲后凭空多出踏脚点即可过。因平台是一次性且位置固定，本处为 set-piece 而非通用机关。

### 5.6 结局序列（R6）
玩家走到最终祭坛→触发 `fourthwall`→依次：HUD 元素逐个掉落/淡出 → 场景元素消解 → 一行收尾文字 → 黑屏。序列为纯视觉，不操作系统窗口。

## 6. 美术方向

### 6.1 视觉风格
简洁、几何、低饱和为底，用高饱和色承载"牺牲相关"的语义信息。风格服从可读性支柱（§1.3.2），不做会干扰"实/虚"判断的复杂纹理。

### 6.2 颜色语义（硬性可读性规则）
- **可穿过/被牺牲**的物体：半透明（默认 alpha 0.35）+ 描边，与实体明确区分。
- **蓝色**专用于 `blue` 概念物体，不得用作纯装饰色，以免误导。
- 若扩展红/黄等彩色门，同理各自专色专用。
- 祭坛用醒目的独立色（当前占位：普通祭坛黄、双槽神龛紫、终点绿）。

### 6.3 角色与动画
- 需求动画：idle、run、jump/fall（可合并）。
- 玩家可见节点用 `AnimatedSprite2D`（或 `Sprite2D`），并将玩家的 `sprite_path` 指向它——重力翻转会自动设置其 `flip_v`，美术无需写代码。
- 动画状态由美术在 `AnimationPlayer`/`AnimatedSprite2D` 中自管，读取 `velocity`、`is_on_floor()` 即可驱动。

### 6.4 概念图标
每个概念一枚图标，需三态可辨（灰/常态/高亮）。MVP 阶段可缺省，用文字 HUD 顶替。

## 7. 音频

### 7.1 BGM
1 首循环 BGM。注意：若日后实现 `sound` 牺牲，BGM 需能被系统静音（作为该牺牲的听觉反馈）。

### 7.2 音效清单（MVP 最小集）
牺牲切换音（开/关各一或复用）、落地音、祭坛触发音。共 2–3 个起步。

### 7.3 牺牲的听觉反馈
听觉是牺牲可读性的一部分：切换概念应有清晰音反馈。`fourthwall` 结局序列中音频应随界面一同解体（逐层淡出/静默）。

## 8. 技术架构

### 8.1 总览
所有会对牺牲做出反应的系统，**只通过 `Sacrifice` 单例的信号通信**。这是架构的根：加新概念、新机关、新关卡，几乎都无需改动核心脚本，只需监听自己关心的 `concept_id`。

### 8.2 Sacrifice 单例 API（权威接口）
注册为 autoload，节点名必须为 `Sacrifice`。

信号：
- `concept_activated(id)` / `concept_deactivated(id)`：某概念效果开/关。
- `concept_unlocked(id)`：某概念变为可切换。
- `slots_changed(new_slots)`：槽位数变化。
- `concept_permanently_sacrificed(id)`：某概念被永久牺牲。

查询：`is_unlocked(id)`、`is_active(id)`、`is_permanently_sacrificed(id)`、`get_active()`、`get_unlocked()`。

命令：`unlock(id)`、`activate(id)`、`deactivate(id)`、`toggle(id)`、`set_max_slots(n)`、`permanently_sacrifice(id)`、`reset()`。

单槽约束与双槽逻辑集中在 `activate()` 与 `set_max_slots()` 中（满槽时弹出最早激活者），不得在别处另写一套。

### 8.3 脚本清单与职责
| 脚本 | 挂载于 | 职责 |
|---|---|---|
| `sacrifice_manager.gd` | autoload `Sacrifice` | 牺牲状态大脑、信号总线、槽位约束 |
| `player_config.gd` | Resource | 手感数值（数据，纯 export） |
| `player.gd` | CharacterBody2D | 移动、重力翻转反应、跳跃禁用 |
| `blue_object.gd` | StaticBody2D | 反应式实体模板（监听某 concept 切碰撞+透明度） |
| `altar.gd` | Area2D | 触发解锁/改槽位/永久牺牲 |
| `sacrifice_input.gd` | Node | 输入动作→概念切换的映射（可在 Inspector 编辑） |
| `hud.gd` | CanvasLayer | 牺牲状态显示；亦是响应 `hud`/`fourthwall` 的对象 |
| `demo_level.gd` | Node2D | 一次性代码测试台（非正式关卡） |

### 8.4 碰撞层约定（硬性）
- **Layer 1 = 世界实体**（地形、墙、机关）。
- **Layer 2 = 玩家**。
- 玩家 `collision_mask` = 1（与世界碰撞）。
- 祭坛/终点 Area 的 `collision_mask` 含 2（检测玩家）。
- 蓝色/反应式实体默认在 Layer 1；被牺牲时由其自身脚本禁用碰撞形状（`set_deferred("disabled", true)`），因此不依赖层切换。

### 8.5 数据驱动
手感全部走 `PlayerConfig.tres`；每帧从 config 重新推导跳跃/重力数值，故运行时修改即时生效。禁止在 `player.gd` 中硬编码任何手感数字。

### 8.6 扩展模式：如何加一个新概念
不改 `sacrifice_manager.gd`。三步：
1. 复制 `blue_object.gd`→改名→改 `concept_id` 默认值→把"献祭时禁用碰撞"换成该概念逻辑（如 `time`：冻结平台/子弹）。想反应的物体挂上即可。
2. 仅当该概念作用于玩家本体（如 `gravity`）时，才需在 `player.gd` 加一小段；作用于世界物体的概念（time/sound/颜色）玩家侧无需改动。UI 类概念则由 `hud.gd` 监听并隐藏对应元素。
3. 放一个 `Altar`（`Action=UNLOCK` 或 `PERMANENT_SACRIFICE`，填 `concept_id`），并在 `SacrificeInput.bindings` 加映射（若为可逆概念需按键）。

单槽/双槽/永久牺牲规则会自动套用到新概念。

### 8.7 已知边界与规避
**在蓝墙内部恢复"蓝"会卡住人物**：切换免费保证了*移动模式*永远能切回，但若人物正嵌在蓝实体内、此时把蓝恢复成实体，会被卡进墙。规避二选一：
- 关卡层面（默认）：不设计"必须在嵌入蓝色时恢复蓝"的谜题。自然走法几乎不会遇到。
- 代码层面（可选加固）：在 `blue_object.gd` 恢复碰撞前用形状查询检测玩家是否重叠，重叠则延迟到玩家离开再恢复。为保持简洁默认不做，需要时再加。

此边界对任何"恢复后重新变实体"的可逆概念同理，关卡设计须知悉。

## 9. 扩展接口（分角色）

### 9.1 关卡设计者
- 地形用 TileMap 或 StaticBody2D，碰撞放 Layer 1。
- 放祭坛：`Area2D`+`altar.gd`+`CollisionShape2D`，Inspector 设 `Action`/`Concept Id`/`Slot Count`，Mask 勾选玩家层（2）。一个祭坛做两件事就叠放两个。
- 放反应式障碍：`StaticBody2D`+`blue_object.gd`（或其复制模板）+ 碰撞形状 + 可见子节点。
- 放终点：`Area2D`，`body_entered` 判断 `body is Player`。
- 必守 §5.4 硬约束。

### 9.2 美术
- 玩家占位块换成 `AnimatedSprite2D`，把玩家 `sprite_path` 指向它，翻转自动生效。
- 机关外观：`BlueObject` 的 `Solid Alpha`/`Passable Alpha` 控透明度；加描边/发光给其可见子节点加特效即可，机制不受影响。
- 遵守 §6.2 颜色语义。

### 9.3 手感微调者
- 只动 `PlayerConfig.tres`，不碰代码（§3.3 参数表）。
- 可为不同角色/关卡建多份 .tres 随时切换。运行时改即时生效。

## 10. 开发计划

### 10.1 MVP 范围（硬边界）
**做**：`gravity`+`blue` 两个可逆牺牲、单槽约束、双槽升级（价格 `jump`）、6 房间关卡（R1–R6）、`fourthwall` 结局、牺牲状态 HUD、基本切换反馈与控制提示。

**不做（jam 期间明确排除）**：战斗 AI/Boss、第 3 个及以上可逆概念、小地图、正经暂停/设置菜单、`hud` 界面坍塌 set-piece（列为 Post-MVP 选做）、储备池概念（friction/time/sound）。

追加顺序（仅当 MVP 提前完成且有余力，按此序）：`hud` 界面坍塌 set-piece → 第 3 个可逆概念（friction，复用性高）→ 更多电波台词/打磨。

### 10.2 实现顺序（vertical slice）
1. 玩家控制器（移动+跳）+ 测试房间 → **先只打磨手感**。
2. `gravity` 牺牲 + HUD 指示。
3. `blue` 系统 + 牺牲。
4. 单槽约束 + 祭坛解锁。（做到此步，核心动作可独立试玩）
5. 双槽升级 + 永久牺牲 `jump`。（点亮银河城回路）
6. 搭 R1–R6 房间与门槛。
7. `fourthwall` 结局序列。
8. 反馈/音效/台词打磨。

### 10.3 里程碑与检查点
- 检查点 ①：核心机制可试玩（完成第 4 步）。
- 检查点 ②：R1–R6 可完整通关（完成第 6 步）。
- 检查点 ③：结局+打磨完成，可提交（完成第 8 步）。

### 10.4 分工建议
程序：§8 架构 + 第 1–5 步。关卡：§5 + 第 6 步 + §5.4 验证。美术：§6 + 角色/图标/机关外观。音频：§7。四方在检查点 ① 后可并行。
